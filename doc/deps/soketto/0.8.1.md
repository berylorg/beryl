# soketto 0.8.1

Verified on 2026-05-10.

## Workspace Use

- `beryl-backend` uses `soketto` for the managed loopback WebSocket protocol boundary to `codex app-server`.
- Beryl enables `default-features = false` and no optional `soketto` features.
- Compression is not selected for the managed app-server boundary because it would add another buffering and failure surface without changing the need to sanitize large generated-image JSON payloads.
- The high-level receive API is not sufficient for Beryl's bounded receive path when it requires a caller-provided `Vec` for a complete message.

## Lifecycle And I/O Notes

- The WebSocket handshake is async and is driven on Beryl backend runtime infrastructure, not on the `gpui` thread.
- The managed client must continue to send `Authorization: Bearer <token>` during the handshake without logging or persisting the raw token.
- Outbound client frames must be masked. Inbound server frames on the loopback client connection must be rejected if they are masked.
- The transport layer owns continuation-frame state, control-frame handling, close handling, reserved-bit rejection, opcode validation, and bounded payload reads before any JSON-RPC method-specific logic runs.
- `tokio-util` compatibility support adapts Tokio I/O to the futures I/O traits used by `soketto`.
- The implemented bridge keeps the public backend session API synchronous and uses a private Tokio runtime call boundary for the async soketto handshake and socket I/O.

## Integration Gotchas

- `connection::Receiver::receive` accumulates a whole message into a caller-owned `Vec`; using it for transcript-history responses would contradict Beryl's bounded hot-path-buffer contract.
- Lower-level frame handling should use `base::Codec::decode_header` and bounded reads of the declared payload length, with `Codec::apply_mask` applied only according to WebSocket masking rules.
- A large generated-image history response can arrive as one WebSocket frame or as fragmented frames, so payload streaming and continuation tracking must not depend on small final messages.
- Control frames can be interleaved with fragmented data messages and must not corrupt the current JSON payload stream.
