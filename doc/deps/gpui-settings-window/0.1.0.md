## Why Beryl Uses It

Beryl uses this crate for app-neutral settings-window mechanics: the preheated settings OS window, broad left navigation, right-pane settings content, row rendering, text and multiline text fields, choice fields, color fields, in-window color picker behavior, row/page action events, page navigation events, and OK/Apply/Cancel/Close events.

## Lifecycle And Event Invariants

- `open_settings_window` creates a reusable GPUI window that may be preheated hidden and later shown with a fresh `SettingsWindowModel`.
- `SettingsWindowHandle::show` and `update_model` replace the presentation model supplied by Beryl; Beryl remains responsible for settings schema, validation, draft state, persistence, and apply/cancel semantics.
- Left sidebar section selection selects the section root page and emits only the app-neutral section-selection signal. It does not also emit a redundant page-navigation request for the implied root page.
- Right-pane navigation rows, breadcrumb targets, and back targets emit page navigation requests with stable host-supplied page ids; the dependency does not decide whether navigation is accepted.
- Navigation rows may also carry row actions; action clicks emit only row-action requests and do not also emit page navigation requests.
- Field edits emit `FieldChanged` with stable field ids and string presentation values.
- Choice rows are app-neutral string-valued field rows with a finite option set supplied by the host. They render as compact dropdown selectors, emit `FieldChanged` with the selected option value, and do not create retained text input state.
- Row actions emit `RowActionRequested` only when the row action exists and is enabled.
- Page actions emit `PageActionRequested` only when the page action exists and is enabled.
- Default and minimum settings-window size are both `800x520` in the current local checkout; host apps may still override through `SettingsWindowOptions`, but Beryl relies on the crate default for the minimal useful supported size.
- Field-row labels and descriptions own the flexible horizontal row space; right-side controls use fixed type-appropriate widths, hug the row's right edge, and do not stretch when the settings window gets wider.
- Field rows reserve an explicit label/control gutter and a label-side minimum width so labels wrap normally instead of being clipped under wide controls.
- Current fixed widths in the local checkout are text `208`, number `96`, multiline `300`, choice `184`, and color text `132` logical pixels, with a `24` logical pixel label/control gutter.
- Field rows may carry one secondary detail field rendered within the same row surface below the primary right-side control. The detail field uses its own stable `SettingsFieldId` and emits ordinary `FieldChanged` events; hosts can use it for compound rows such as a source selector with a concrete value editor that collapses when no concrete value is active. In split detail panes, the primary control is top-aligned with the label while the secondary detail field stays in the same right-side control column.
- `SettingsFieldKind::Number` is a single-line string field with compact numeric presentation width for short values such as 6 to 7 digit settings.
- Field rows render a right-side action cluster only when actions exist, so actionless text and multiline fields do not lose width to an empty cluster.
- Action-bearing single-line text rows render as one right-aligned control column with the fixed-width text field above the action cluster. Beryl's Notifications sound-picker row relies on this generic text-plus-actions shape so long file names and `Choose...`/`Clear` actions do not force the label into one-character wrapping at the supported minimum width.
- Color picker popups bound their saved-colors swatch grid to an internal scroll region so large Beryl theme-derived saved color sets do not make the popup exceed the supported default settings-window height.
- Navigation rows use the crate-owned right-facing thick triangle `▸`, paired with Beryl's down-facing selector triangle family rather than ASCII `>` or a host-provided suffix.
- Pages may carry an optional page-local split. The split belongs to the selected page body, is separate from the broad settings sidebar, and emits `PageSplitItemSelected` with host-supplied page and item ids.
- Page-local split items are app-neutral. They may include selected state, subtext, and compact preview style metadata, but the dependency does not know about Beryl theme roles or theme inheritance.
- Page-local split lists in the local Phase 20 checkout are fixed-height bounded render surfaces: long lists render only a visible row window plus overscan while preserving total scroll extent, selected item reveal, stable item ids, and a scroll state independent from the selected page's detail rows.
- Page-local split detail rows use a compact top-aligned field-row layout at the supported minimum width, with the label stack as a narrower flexible left column and the fixed-width control cluster right-aligned at the top of the same row.
- Retained text-field widgets and latest-known color values are keyed by field id, avoiding repeated linear retained-state lookups during model synchronization.
- The local Phase 22 checkout exposes app-neutral, content-free settings-window diagnostics through `SettingsWindowHandle::diagnostics_snapshot`, `SettingsWindowView::diagnostics_snapshot`, and `SettingsPanel::diagnostics_snapshot`. The snapshot reports selected page/section ids, detail-row counts, split-list total/rendered counts and visible range, render-tree timing, model-sync timing, option-sync timing, input-sync counts, color-preview lookup counts, color-model lookup counts, and a dominant timing category. It does not expose setting labels, setting values, file paths, validation text, or host theme documents.
- Disabled row and page actions remain visible in the model, may carry an optional disabled reason, and do not emit their normal request events.
- Color picker popups are transient settings-window state. Beryl may observe or close them through the handle without applying, canceling, or hiding the settings window.
- Hiding the preheated window clears transient popups so stale color-picker state is not retained across later shows.

## Phase 8 API Coverage

The local Phase 8 dependency checkout adds:

- `SettingsPageId`, `SettingsPage`, and selected-page support on `SettingsWindowModel`
- section root pages plus ordered subpages
- breadcrumb metadata and optional back targets
- `SettingsPageActionId`, `SettingsPageAction`, and page action request events
- row kind support for editable field rows, navigation rows, and action-only rows
- host-supplied modified indicators on rows and pages
- enabled/disabled state plus optional disabled reasons for row and page actions
- crate-owned right-facing chevron rendering for navigation rows
- row-action rendering on navigation rows without replacing the crate-owned chevron
- validation for duplicate page ids, duplicate page action ids, invalid selected pages, missing navigation targets, missing breadcrumb targets, and missing back targets
