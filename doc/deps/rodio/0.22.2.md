# rodio 0.22.2

Verified: 2026-05-06

Resolved versions:

- `rodio` 0.22.2
- `cpal` 0.17.3
- `hound` 3.5.1

Beryl uses `rodio` for best-effort short WAV notification playback on Windows, off the GPUI thread.

## Enabled Features

Beryl enables `rodio` with `default-features = false` and these features:

- `playback`
- `hound`

`playback` enables CPAL-backed output. `hound` enables the pure-Rust WAV decoder path. The default Symphonia decoder stack is not enabled.

`beryl-app` depends on `rodio.workspace = true` only under `cfg(target_os = "windows")`.

## Invariants And Gotchas

Keep a `MixerDeviceSink` alive while playback should remain possible; dropping it stops audio.

For Beryl's serialized notification queue, a worker-owned `MixerDeviceSink` plus one persistent `Player` is the integration shape. The worker appends one decoded source to that persistent player and blocks on `Player::sleep_until_end()` on the worker thread.

Avoid creating and dropping a `Player` for each notification sound. `Player` drop stops its sounds, and the first cold Windows playback can be truncated when the per-sound player reaches `sleep_until_end()` and drops while the OS audio stream is still settling. Keeping the player alive with the stream preserves the first playback path and still serializes each notification source through the worker queue.

Beryl keeps the worker-owned sink open across notification attempts, records the CPAL default output device id used to open the sink, and checks the current default output device id before each playback attempt. When the known current default id differs, or when Rodio's stream-error callback marks the stream stale, Beryl lazily reopens the default output sink before playback. This keeps normal notification latency low without keeping a known-failed stream forever.

Call `MixerDeviceSink::log_on_drop(false)` if Beryl owns diagnostics, because without `rodio`'s `tracing` feature the default drop path prints to stderr.

With `hound` enabled and `symphonia-wav` disabled, `Decoder` uses `decoder::wav::WavDecoder`. `Decoder::new_wav(data)` sets a WAV hint, but the hound path still probes the stream with `hound::WavReader::new`, rewinds to the original stream position, and reports `DecoderError::UnrecognizedFormat` when no enabled decoder accepts the data.

The hound WAV decoder supports 32-bit float samples and integer 8-bit, 16-bit, 24-bit, and 32-bit samples. Unsupported float bit depths and integer bit depths above 32 end iteration after rodio logs or prints a diagnostic, so Beryl must treat decode and playback failures as non-fatal.

Rodio output is backed by `cpal::Stream`. `DeviceSinkBuilder::from_default_device()` queries CPAL's default output device once. `DeviceSinkBuilder::open_default_sink()` opens the default output device, builds a stream, starts it with `StreamTrait::play()`, and returns a `MixerDeviceSink` that owns the stream. Open sinks do not automatically follow later OS default-device changes, so Beryl must compare `DeviceTrait::id()` values and reopen when it wants notification playback to follow the current default device.

Blocking file open, metadata read, decoder construction, output-device probing, stream open, and `Player::sleep_until_end()` must stay off the GPUI thread.

On Windows, CPAL's default host is WASAPI. The default output device can be absent, disconnected, invalidated, or fail config/build/play operations. CPAL initializes COM per thread for WASAPI and tolerates `RPC_E_CHANGED_MODE` when another library already initialized COM differently. Keep audio work on a dedicated worker thread and treat initialization failures as diagnostics.

The WASAPI stream owns a dedicated `cpal_wasapi_out` thread. Dropping the stream sends a terminate command and joins that thread.
