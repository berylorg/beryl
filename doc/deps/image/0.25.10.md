# image 0.25.10

Verified on 2026-05-12.

## Workspace Use

- `beryl-app` uses `image` directly to decode transcript media dimensions before handing compressed bytes to `gpui::Image::from_bytes`.
- `gpui` also depends on `image` and decodes `gpui::Image` values into renderable `RenderImage` assets.
- The workspace manifest requests `image = "0.25.1"`; `Cargo.lock` resolves `image 0.25.10`.
- Enabled features through Beryl and GPUI: `default`, `default-formats`, `rayon`, `avif`, `bmp`, `dds`, `exr`, `ff`, `gif`, `hdr`, `ico`, `jpeg`, `png`, `pnm`, `qoi`, `tga`, `tiff`, and `webp`.

## Beryl Entry Points

- `crates/beryl-app/src/shell/transcript_media/load.rs` calls `image::load_from_memory_with_format(bytes, image_format(format))` in `decoded_raster_dimensions`.
- The same load path then calls `Image::from_bytes(format, bytes.clone())`; Beryl retains the compressed byte vector separately as `Arc<[u8]>` in `TranscriptMediaLoadedImage`.
- Beryl accepts raster transcript media formats mapped from paths: PNG, JPEG, WebP, GIF, BMP, and TIFF.
- Beryl decodes native generated inline results from base64 into a PNG byte vector before this same path.

## Dependency Allocation Behavior

- `load_from_memory_with_format` wraps the input slice in an `io::Cursor`, then calls `free_functions::load`, `ImageReader::decode`, and `DynamicImage::from_decoder`.
- `ImageReader::decode` constructs a decoder, calls `limits.reserve(decoder.total_bytes())`, applies limits to the decoder, then materializes a `DynamicImage`.
- `Limits::default()` has no width or height limit and a default `max_alloc` of 512 MiB. This is per decode attempt, not a process-wide retained-memory budget.
- `DynamicImage::from_decoder` reads the complete decoded raster into a new owned `ImageBuffer` through `decoder_to_vec`.
- Beryl drops the `DynamicImage` after reading `width()` and `height()`, so this first decode is transient CPU/memory work unless the allocator retains freed heap pages.
- GPUI later decodes `gpui::Image` values again for rendering. Non-animated images are converted to RGBA and then BGRA, producing one owned decoded frame. GIF and animated WebP decode and retain all frames in a `SmallVec<Frame>`.

## Caches And Globals

- The `image` crate itself does not cache decoded rasters or compressed inputs on Beryl's normal `load_from_memory_with_format` path.
- `image::hooks` has process-global `RwLock` registries for custom decoding and format-detection hooks: `DECODING_HOOKS` and `GUESS_FORMAT_HOOKS`. Beryl did not show use sites for registering hooks.
- `image::guess_format` uses a static magic-byte table and no dynamic cache.
- Enabling `rayon` exposes parallel buffer APIs and enables AVIF/exr threading features, but the Beryl/GPUI decode paths inspected do not call `image` parallel iterators directly. Any rayon global thread-pool memory is owned by `rayon`, not by `image` decoded-image caches.
- GPUI keeps `ImageDecoder`/`ImageAssetLoader` results in `App.loading_assets`, keyed by asset type and source hash, until `remove_asset` is called. For Beryl transcript `Arc<Image>` sources, this can retain decoded `RenderImage` frames independently of Beryl's `TranscriptMediaCache`.

## Retention Boundaries For Beryl

- The dependency provides only a per-decode default allocation limit, not a deterministic retained-memory limit across cached images.
- Beryl must bound compressed image bytes retained in `TranscriptMediaCache`; the current cache is entry-count bounded, not byte-count bounded.
- Beryl must account for decoded render memory retained by GPUI assets. The rough lower bound is `width * height * 4` bytes per non-animated frame; animated formats can retain that per frame.
- Beryl should remove GPUI assets for images evicted or invalidated from its own media cache, or avoid using long-lived GPUI asset cache entries for transcript images that Beryl considers evicted.
- Beryl should reject or downscale transcript media before `Image::from_bytes` if decoded dimensions exceed an application-owned memory budget.
