## Lifecycle And Threading Notes

- `gpui` owns the UI event loop and window lifecycle once the application shell is live.
- `Window::on_window_should_close` callbacks may cancel native window destruction by returning `false`; Beryl uses this for the main shell close path so the app event loop quits before the shell window is destroyed.
- `Subscription` cancels its callback when dropped; long-lived app observers must be stored or explicitly detached.
- Blocking filesystem, process, and protocol work must stay off the `gpui` thread, which matches the workspace design contract in `doc/design.md` and `crates/beryl-app/doc/design.md`.
- `Context::spawn` supplies a borrowed `AsyncApp`; async tasks that outlive the call site must clone that context before awaiting.
- `App::defer`, `Window::defer`, and `Context::defer_in` schedule work for the end of the current effect cycle so entities and windows currently on the stack can be returned before the callback runs.
- `WeakEntity::update` and `Entity::update_in` let Beryl drive later repaint-triggering state updates from async timers and cross-entity event handlers without calling layout-only window APIs out of phase.
- `gpui::list` is intended for differently sized items and caches measured item heights. When an item's height changes outside normal visible-area updates, the owning view must notify the list via `ListState::splice` or `ListState::reset`.

## Integration Gotchas

- Upstream `gpui` 0.2.2 enables `font-kit`, `wayland`, `x11`, and `windows-manifest` by default. Beryl disables upstream defaults and enables only `windows-manifest`.
- GPUI `test-support` enables additional dev/test dependencies and platform features such as `wayland` and `x11`; keep it in dev-dependencies rather than normal reusable-crate dependencies.
- The local GPUI fork makes the HTTP client dependency optional behind the `http-client` feature. Beryl's normal workspace resolution enables only `windows-manifest` for `gpui`; `cargo tree --workspace --edges features -i gpui` showed no `gpui/http-client` activation.
- Because `gpui/http-client` is not enabled in Beryl's normal workspace build, `zed-reqwest` and `ring` are not present in the normal/build dependency tree. `cargo tree --workspace --edges normal,build -i zed-reqwest` and the same command for `ring` both returned package-not-found.
- GPUI `test-support` in the local fork no longer enables `http-client`; HTTP-specific GPUI test helpers require the explicit `http-client-test-support` feature.
- Published `gpui 0.2.2` does not expose the local fork's `App::should_reduce_motion` or `App::caret_blink_interval` APIs.
- Activity-caret blink scheduling remains Beryl-owned and uses official GPUI async timer APIs. Beryl owns platform caret policy directly in `crates/beryl-app/src/shell/render/transcript/activity_caret.rs`.
- On Windows, Beryl reads text-caret blink policy through `GetCaretBlinkTime`; `0` and `u32::MAX` mean disabled blinking. Non-Windows currently uses Beryl's conservative 530 ms fallback.
- `Div::key_context` stores a `KeyContext` on the active dispatch-tree node during paint. Key binding resolution uses the focused node's dispatch path and context stack, so bindings can match focused descendants of the element that declares the context.
- `KeyBinding::new` binds a keystroke to an action within an optional context. Deeper matching contexts take precedence over ancestor contexts, and later-added bindings win at the same depth.
- `InteractiveElement::on_action` registers action listeners on the active dispatch-tree node. Bubble-phase action handlers stop propagation by default unless they call `cx.propagate()`.
- `Window::on_key_event` registers a low-level key listener on the active dispatch-tree node. Raw key listeners are useful for popup traversal and dismissal, but context-scoped key bindings plus actions are the direct route for focused input commands such as composer submission.
- `StatefulInteractiveElement::on_scroll_wheel` is filtered through `hitbox.should_handle_scroll(window)`, so overlay widgets that must block underlying scroll targets should occlude the area they cover rather than relying on hover semantics.
- `ListState::set_scroll_handler` is the right hook when Beryl needs post-scroll notifications from `gpui::list` without replacing the list's built-in wheel handling.
- `Window::request_animation_frame` is the clean repaint hook for view-local animation in official GPUI.
- `ClipboardItem::text()` concatenates only string entries and ignores image entries. Composer image paste must inspect `ClipboardItem::entries()` and match `ClipboardEntry::Image` before falling back to text.
- `ClipboardItem` exposes entries for inspection and owned iteration, but its public constructors cover single string entries, string entries with metadata, JSON metadata on a string entry, or a single image entry. Beryl can serialize custom editor selections such as composer image atoms as external plain text with `ClipboardItem::new_string_with_json_metadata("[Image A]".to_string(), metadata)`, then inspect `ClipboardItem::metadata()` or `ClipboardString::metadata_json()` on paste.
- `ClipboardItem::new_image(&Image)` stores `ClipboardEntry::Image(image.clone())`, and `Image` owns `bytes: Vec<u8>`. Beryl can avoid full image byte clones in transcript render-time menu targets and save-action state by keeping its own `Arc<[u8]>`, but the final GPUI clipboard handoff currently clones the GPUI image payload through the public API.
- GPUI 0.2.2 does not expose a cross-platform public API for writing multiple clipboard representations or custom MIME formats for one logical selection. Beryl should not rely on placing both image bytes and explanatory text into one portable clipboard item for composer atom copy; image bytes should stay in Beryl-owned process memory and clipboard metadata should contain only an opaque token.
- On Windows, GPUI clipboard reads choose one best-match entry. Clipboard images are exposed as `Image { format, bytes, id }`; file drops from the clipboard are exposed through string metadata, not image entries.
- On Windows, GPUI writes string metadata through private clipboard formats named `GPUI internal text hash` and `GPUI internal metadata`. `read_string_from_clipboard` returns metadata only when the stored hash still matches the visible `CF_UNICODETEXT` text, so Beryl should still validate visible clipboard text against any token-backed composer payload before restoring atoms.
- On Windows, GPUI clipboard writes use only the first entry in a `ClipboardItem`, so Beryl-owned clipboard writes that need deterministic external paste behavior should write the exact plain text representation they want external applications to receive.
- On X11, GPUI keeps the most recent owned `ClipboardItem` in process and returns it with metadata while the GPUI client still owns the clipboard; clipboard manager or external-owner round trips may return only plain text. Beryl composer paste must therefore treat metadata restoration as best-effort and use ordinary text paste when metadata is absent.
- `Image::from_bytes(format, bytes)` creates a GPUI image asset with a stable content-derived id. `img(Arc<Image>)` renders those bytes through GPUI's image cache, and `StyledImage::object_fit(ObjectFit::Contain)` fits the decoded image inside fixed popup bounds without stretching the aspect ratio.
- `App::prompt_for_new_path(directory, suggested_name)` opens a native save destination picker and returns `oneshot::Receiver<anyhow::Result<Option<PathBuf>>>`. Picker cancellation resolves to `Ok(None)`. Picker-open failures resolve to `Err`, especially on Linux when the portal cannot open.
- The save picker supports an initial directory and suggested file name. It does not expose public file-type filters or a public default-extension option; callers that need an extension must include it in the suggested name and decide how to handle extensionless selected paths.
- The Windows save picker uses `IFileSaveDialog`; macOS uses `NSSavePanel`; Linux uses the xdg desktop portal save request. The API is public through `App` and re-exported by `gpui`.
- `EventEmitter<E>` is a marker trait for typed entity events. A view that implements it may call `cx.emit(event)`, and another entity can subscribe through `cx.subscribe(&emitter, handler)`. Beryl uses this for composer input marker-click events without making the shared text input depend on shell types.
- `Window::handle_input` must be called from an element's `paint` method with the focused `FocusHandle`; the registered input handler is frame-scoped and must be refreshed by painting.
- GPUI text input protocol ranges are UTF-16, while text layout and Beryl text models use UTF-8 byte offsets. Reusable text inputs must convert explicitly at the GPUI boundary and keep model operations on valid UTF-8 boundaries.
- `TextSystem::shape_line` is only suitable for single-line text. Reusable multiline input should use `TextSystem::shape_text` with a wrap width and use `WrappedLine` plus `wrap_boundaries` for visual-line geometry.
- Key dispatch tries action bindings first. If propagation remains active and the keystroke has a text character, GPUI forwards text to the active input handler. Text widgets should handle editing commands as actions and let ordinary typed characters arrive through `EntityInputHandler::replace_text_in_range`.
- `InteractiveElement::on_action` handlers stop propagation by default. A reusable text widget should call `cx.propagate()` only when a recognized action is intentionally left to host-owned behavior.

### Windows Window Show And Focus Reentrancy

- GPUI platform callbacks for request-frame, active-status, and input events enter window updates through app-borrowing paths. If a synchronous native Windows call reenters those callbacks while a GPUI update already owns the mutable app borrow, GPUI logs the failed borrow through `.log_err()`, which can appear as `RefCell already borrowed`.
- `Window::activate_window` is the public GPUI activation API. On Windows, GPUI schedules foreground activation work through its executor and uses `ShowWindowAsync(SW_RESTORE)` only when the target window is minimized.
- Code that owns raw Windows show/hide for a GPUI window should avoid synchronous `ShowWindow` from inside a GPUI window/entity update. Prefer `ShowWindowAsync` or a non-activating show command, then defer GPUI activation and widget focus with `Context::defer_in` or an async task after the current update unwinds.
- `WindowOptions::focus` may activate a window during creation. If a caller also needs app-owned field focus, creating the window unfocused and deferring activation plus field focus avoids a duplicate focused-open path.

### Windows Rendering Memory Baseline

- On Windows, `Application::new()` initializes the GPUI platform stack before Beryl application data is loaded. That stack includes OLE/windowing setup, DXGI/D3D11 device creation, DirectWrite text setup, executors, hidden helper windows, and GPU-driver modules.
- Opening a GPUI window creates renderer resources separate from Beryl-owned Rust data. The Windows backend uses DirectX renderer, DirectX atlas, DirectComposition/window resources, DirectWrite text rendering, swapchain-related resources, and vendor GPU driver allocations.
- GPU process-memory counters are useful for classifying this baseline, but their values must not be treated as Rust heap ownership. Process Private Bytes, Working Set, mapped module image size, and `\GPU Process Memory(*)\Total Committed` are related but not interchangeable.
- `ProcessModule.ModuleMemorySize` helps identify loaded rendering, text, shell, and GPU-driver modules. It is mapped image footprint and not a private-heap attribution mechanism.
- Release-profile Beryl runs are the relevant baseline for product memory investigation. Debug builds can carry additional GPUI debug and inspector paths, and the Windows renderer source probes D3D/DXGI debug support under debug assertions.
- Precise heap, GPU allocation, thread-stack, and mapped/private region attribution is not available from GPUI public APIs or ordinary PowerShell process counters. That level of attribution requires external profiling tools or additional targeted instrumentation.
- GPUI Windows atlas source in `platform/windows/directx_atlas.rs` uses a default atlas size of `1024 x 1024` and maximum atlas size of `16384 x 16384`. Monochrome glyph atlases use `DXGI_FORMAT_R8_UNORM`, while polychrome image/emoji atlases use `DXGI_FORMAT_B8G8R8A8_UNORM`.
- GPUI Windows renderer source in `platform/windows/directx_renderer.rs` uses a triple-buffered BGRA swapchain and full-window BGRA path intermediate textures, including a multisampled path target with `PATH_MULTISAMPLE_COUNT = 4`. On the measured `1560 x 1140` device-pixel window, local texture diagnostics estimated about `54 MiB` for swapchain plus path resources. Those resources explain the visible-window floor, not the additional normal-shell glyph/atlas delta.
- For memory profiling, distinguish atlas page growth, full-window path/swapchain resources, and driver/runtime allocations triggered by glyph atlas activity. Atlas page bytes are roughly `width * height * bytes_per_pixel` per page, but Phase 8 diagnostics showed the normal shell can add about `153 MiB` over `shell-minimal-render` while directly creating only one additional `1 MiB` monochrome atlas page.
- The Phase 8 follow-up entrypoints for Windows memory are `DirectXAtlas::get_or_insert_with`, `DirectXAtlasTexture::upload`, `ID3D11DeviceContext::UpdateSubresource`, `Window::paint_glyph`, `TextLayout::paint`, and DirectWrite glyph rasterization. The direct `CreateTexture2D` call sites alone are insufficient to explain the measured normal-shell excess.
- `DirectXAtlas::get_or_insert_with` checks the atlas cache, rasterizes on miss, allocates one tile, and uploads that tile. `DirectXAtlasTexture::upload` performs `UpdateSubresource` for a tile rectangle, so many first-frame atlas misses can mean many tiny D3D texture updates even when the process creates only one atlas page.
- Phase 9 diagnostics measured this directly: baseline normal shell performed `275` monochrome atlas uploads totaling only `37,154` bytes, while `shell-minimal-render` performed no atlas uploads and stayed about `154 MiB` lower in Private Bytes. `minimal-text-window` performed `5` uploads totaling `530` bytes and landed around `68.5 MiB`; `shell-textless-normal-geometry` performed `227` uploads totaling `32,298` bytes and still landed around `210.3 MiB`.
- Phase 10 confirmed the GPUI Windows fix target is upload-call policy, not atlas page size or payload byte retention. A diagnostic batcher kept the same `1,321` glyph paint calls, `275` atlas misses, and `275` tile insertions as baseline, but reduced atlas uploads from `275` calls to `2` calls. Private Bytes dropped from `216.836 MiB` to `66.949 MiB`.
- Batching must retain existing atlas contents. The invalid first diagnostic attempted full-page upload from a sparse dirty buffer and could erase previously uploaded glyphs, causing missing-letter artifacts. A correct implementation needs a persistent CPU mirror, tight dirty rectangles, or another representation that never uploads zeros over valid atlas regions.
- The production GPUI Windows fix should batch per-tile glyph uploads into one upload per dirty atlas texture or dirty region before renderer use, or otherwise avoid issuing hundreds of tiny `ID3D11DeviceContext::UpdateSubresource` calls during startup text paint.
- The formal fork patch in `gpui-fork` implements the production direction by keeping a CPU-side mirror per DirectX atlas texture, accumulating dirty bounds, and flushing the dirty rectangle before `DirectXAtlas::get_texture_view` returns the shader resource view. Fresh production-fork Beryl startup measured `66.859 MiB` Private Bytes and `42.551 MiB` VMMap `Private Data`.
- The CPU-side mirror intentionally trades a small amount of ordinary process memory for avoiding hundreds of tiny D3D atlas uploads. At the default `1024 x 1024` atlas size this adds about `1 MiB` for a monochrome glyph atlas and about `4 MiB` for a BGRA color atlas. This is acceptable for the measured startup path, but it is not a general bound for all atlas growth: GPUI's Windows atlas code can grow textures up to `16384 x 16384`, where a full BGRA CPU mirror would be about `1 GiB`. Future fork polishing should either cap mirror-backed atlas growth, switch large atlas pages to a sparse dirty-region representation, or otherwise avoid unbounded full-page CPU mirrors for large color atlases.

### Phase 3 Dependency Memory Retention Audit

- `App::loading_assets` is a process-lifetime `FxHashMap<(TypeId, u64), Box<dyn Any>>` of shared asset tasks. `App::fetch_asset` removes and reinserts a `Shared<Task<_>>`, so completed asset results remain cached until `App::remove_asset` is called. There is no item or byte cap. Beryl can remove an asset by source through GPUI public APIs, but `Image::remove_asset` removes only the asset-task entry; already uploaded sprite-atlas tiles are not removed unless Beryl also holds or obtains the `Arc<RenderImage>` and calls `App::drop_image`.
- `RetainAllImageCache` is not an LRU cache despite its source comment. It is a `HashMap<u64, ImageCacheItem>` with no count, byte, decoded-pixel, or task cap. Its explicit lifecycle controls are `clear`, `remove`, and entity release, which call `App::drop_image` for loaded images. Beryl source inspection did not find `retain_all`, `image_cache`, or `.image_cache(...)` use sites, so Beryl's current `img(Arc<Image>)` paths use GPUI's app asset cache rather than this element cache. Beryl now calls `Image::remove_asset` when transcript media entries or transcript image-preview popup images are evicted or closed, but this only removes GPUI asset tasks; deterministic uploaded atlas byte reclamation still requires a GPUI-fork lifecycle or atlas-budget change.
- GPUI image decoding retains full decoded BGRA frames in `RenderImage`. GIF and animated WebP sources retain every decoded frame. `Image::from_bytes` owns encoded bytes and keys the GPUI asset cache by a content-derived id, so repeated render construction for identical bytes should hit the same GPUI asset entry, while distinct image bytes grow the app asset cache. `ImageAssetLoader` and `ImageDecoder` have no resize, frame-count, pixel-count, or byte limit.
- The per-window sprite atlas retains glyphs, SVG alpha masks, and images. On Windows, `DirectXAtlas` creates pages at least `1024 x 1024` and at most `16384 x 16384`, with one byte per pixel for monochrome pages and four bytes per pixel for polychrome pages. There is no total page count or total byte cap. The fork's CPU mirror is one full `Vec<u8>` per atlas page.
- `DirectXAtlas::remove` decrements a page-level live-key counter and drops a whole texture only when that page becomes unreferenced. It does not deallocate or reuse an individual removed tile while other keys remain live on the same page. `Window::drop_image` removes image frame keys, but GPUI exposes no equivalent public cleanup path for glyph or SVG atlas keys. Device loss and window drop clear the atlas.
- `TextSystem` has app-lifetime maps for font resolution, font metrics, and glyph raster bounds. `raster_bounds` is keyed by font id, glyph id, font size, subpixel variant, scale factor, and emoji flag. These maps have no count or byte cap and no public clear operation; Beryl can only limit churn by limiting distinct fonts, sizes, scale factors, and glyphs used during the app lifetime.
- `WindowTextSystem` owns a per-window `LineLayoutCache` with current and previous frame maps for unwrapped and wrapped line layouts. `finish_frame` swaps frames and clears the old current maps, so this is a two-frame layout cache rather than an app-lifetime text-history cache. It has no explicit byte cap, but stable UI text should plateau after the first rendered frames.
- `TextSystem::wrapper_pool` retains returned `LineWrapper` instances by `(FontId, font_size)`. Each wrapper has fixed ASCII width slots and an unbounded `HashMap<char, Pixels>` for non-ASCII character widths. The retained pool is bounded only by distinct font/size keys and peak concurrent wrapper handle count.
- `DirectWriteTextSystem` retains selected fonts in `fonts`, `font_selections`, and `font_id_by_identifier`. Custom fonts added through `add_fonts` update the DirectWrite custom collection, but GPUI exposes no font removal or cache-clear API. Color glyph rasterization creates temporary Direct3D layer textures, render-target/staging textures, and a constant buffer per rasterization call; those are not persistent caches, and persistent color glyph memory is the atlas tile uploaded afterward.
- `DirectXRenderer` owns fixed window resources plus high-water pipeline buffers. Swap chains use `BUFFER_COUNT = 3`. Path rendering owns one BGRA intermediate texture and one BGRA MSAA texture with `PATH_MULTISAMPLE_COUNT = 4`, sized to the current window and recreated on resize. Per-pipeline structured buffers start at fixed item counts (`shadow=4`, `quad=64`, `path_rasterization=32`, `path_sprite=4`, `underline=4`, `monochrome_sprite=512`, `polychrome_sprite=16`) and grow to `next_power_of_two` of the largest scene batch seen; they do not shrink until renderer drop or device-loss recreation.
- Window/app scratch structures clear after frames but generally retain allocation capacity: the thread-local element arena starts at `1 MiB` chunks and keeps any chunks added for a large frame; `Frame::clear`, `Scene::clear`, `layout_id_buffer.clear`, and `TaffyLayoutEngine::clear` clear contents but do not provide deterministic byte caps. First-window rendering can therefore look like growth from startup even when it is the fixed high-water allocation for a stable shell.
- Fixed startup cost that should not be mistaken for live-session growth includes `WindowsPlatform::new` OLE setup, the hidden platform message window, shared DXGI/D3D11 device/context creation, DirectWrite setup, the first visible `DirectXRenderer`, shader/pipeline creation, swap-chain/path textures, first-frame text shaping, first atlas pages, and renderer buffer warm-up. A Phase 4 measurement should separate post-first-idle plateau from repeated creation of new image bytes, fonts, glyph sizes, SVG sizes, or peak-size scenes.
